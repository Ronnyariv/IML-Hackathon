# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

An IML (Intro to Machine Learning) hackathon for predicting bike-sharing demand. The task is to predict the number of rides starting from a given station during a given hour (`demand`), given station metadata, weather, and calendar features. The dataset covers three cities (`city 1`, `city 2`, `city 3`).

## Commands

**Prepare local train/validation split** (from raw `dataset/train_set.csv`):
```bash
python split_data.py
# Outputs: dataset/local_train_set.csv, dataset/local_validation_set.csv, dataset/city3_generalization.csv
```

**Train your model** (run from within the submission folder):
```bash
cd submissions/my_team
python train.py
# Outputs: submissions/my_team/weights.joblib
```

**Check submission format** (validates all teams, or one specific team):
```bash
python check_submission_format.py
python check_submission_format.py my_team
```

**Run evaluation** (instructor-side, requires `dataset/public_test_targets.csv` and `dataset/private_test_labels.csv`):
```bash
python evaluate.py
python evaluate.py --eval_dir dataset --submissions_dir submissions --output_csv mae_by_city.csv
```

**Build test evaluation files from raw rides** (instructor-side):
```bash
python build_station_hour_eval_data.py --input_csv dataset/test_set.csv
# Outputs: dataset/public_test_targets.csv, dataset/private_test_labels.csv
```

## Architecture

### Data flow

1. **Raw data** (`dataset/train_set.csv`): ride-level CSV. Each row is one bike ride with `started_at`, `ended_at`, station metadata, weather features, and calendar columns.

2. **`split_data.py`**: splits the raw training data into `local_train_set.csv` / `local_validation_set.csv` (by time: last 7 days per city become validation), and separates `city3_generalization.csv` for out-of-distribution testing.

3. **`submissions/my_team/preprocessing.py`**: transforms ride-level data into station-hour demand format:
   - `clean_rides()` — drops null stations/coords, filters invalid ride durations, removes ride-only columns.
   - `aggregate_demand()` — counts rides per `(city, station, hour_ts)` for hours 6–22.
   - `compute_feature_medians()` / `fix_features()` — imputes missing numeric station/weather features with training-set medians.
   - `build_features()` — constructs the feature matrix used by LightGBM: time features, city one-hot, station historical means, base numerical columns, and interaction terms (e.g. `rush_hour_workday`, `rain_x_rush_hour_workday`).

4. **`submissions/my_team/model.py`** (`BikeDemandModel`): stateless class that holds loaded artifacts (`model`, `medians`, `station_means`, `city_means`, `feature_columns`) and runs inference via `build_features()` → LightGBM predict → clip to ≥ 0.

5. **`submissions/my_team/train.py`**: end-to-end training script. Reads `local_train_set.csv`, calls preprocessing, builds station/city historical means, trains LightGBM, and saves all artifacts as `weights.joblib`.

6. **`submissions/my_team/predict.py`** (`Model(BaseModel)`): thin evaluator-facing wrapper. Loads `weights.joblib` with joblib, calls `BikeDemandModel.load_artifacts()`, and delegates `predict()`. **Do not edit this file.**

### Submission contract

The evaluator (`evaluate.py`) expects every team folder under `submissions/` to contain:
- `train.py` — training script with a `main()` function
- `model.py` — must define `BikeDemandModel` (not `Model`)
- `predict.py` — must define `class Model(BaseModel)` that imports `BikeDemandModel` from `model` and delegates to it
- `weights.joblib` — pre-trained artifact

The evaluator calls `Model().load(weights_path)` then `Model().predict(test_df)`. `test_df` is station-hour format (one row = one city+station+hour to predict), never raw ride-level data. Predictions must be a 1D non-negative numeric array with length equal to `len(test_df)`.

`base_model.py` at project root defines the abstract `BaseModel` interface; it is importable by all submission folders because `evaluate.py` inserts the project root into `sys.path` at runtime.

### Feature engineering key points

- **Most powerful feature**: `station_hour_mean` — historical average demand for `(city, station, hour_of_day, day_of_week)`. Falls back to `city_hour_mean`, then 1.0 for unseen stations.
- **Evaluation hours**: 6:00–22:00 only (hours outside this range are filtered before aggregation and never appear in test targets).
- Station ID normalization: floats like `31631.0` and strings like `'31631.0'` are normalized to `'31631'` by `build_station_hour_eval_data.py`, so models must handle heterogeneous station ID formats in `test_df`.

### Evaluation metric

**MAE** (mean absolute error) on clipped predictions (`max(0, prediction)`), reported per city and overall. Lower is better.
